QA Testing Procedures

Updated for Flare V11 - Jay Wulkowicz 1/21/2015

Process to follow a Feature from development to finished/shelved.

 

  1. Move your name to a Trello card that has been moved to Ready For Testing
    • Verify that there is a Feature in the bug base (TFS) for this Feature and the Trello card links to it
    • If no Feature exists, one needs to be created and then linked to the Trello card
  2. Move the card over to the Testing column in Trello
  3. Check the Release Notes webpage for detailed information on the new Feature and for testing ideas to get you started
    • This will be more up-to-date than appnews!
  4. Test the feature. Document if time permits. (Update or create a new test plan)
    • Work with the assigned developer with any questions or concerns
  5. Link all bugs to the Feature in bug base (TFS).
  6. Move the card back to In Development if the feature has outstanding issues. That means:
    • If you cannot proceed due to a critical bug
    • or if you have tested everything you can, but bugs remain and the feature cannot be shelved
  7. Email the Developer and QA lead if the card is moved back to In Development
  8. If the feature has no more bugs and can be shelved, move the Trello card over to Finished
  9. Once you have updated/created a test plan for the feature, mark the card with the appropriate label

 

Process for FFT

  1. Move your name to a Trello card under "Modules for FFT"
  2. Move the card over to the Testing column in Trello
  3. Find and execute the associated Test Plan(s)
    1. Either use the QA Test Plan Project
    2. or the published output on maddev
  4. Most test plans are not 100% up-to-date. Use your knowledge of the app to cover the gaps. Helpful tips:
    1. Basic components of a test plan are:
      1. UI
      2. Editor
      3. Output (all)
      4. Baseline
    2. How does your module impact new features
    3. How do new features impact your module
  5. If you find a bug:
    1. Compare to previous version. 
      1. If same, write it up and kick it Previous Version. Let us know.
      2. If new, write it up and let us know
  6. When done with a module:
    1. Move the card over to the Done column
    2. Update the time on the card if needed. Turn on the green label on the card.
    3. If there were bugs that must be fixed, turn the red label on the card.

 

Process for Baseline and Regression testing

  1. Launch the appropriate query for the baseline/regression
  2. Select some bugs and assign them all to you
  3. When finished regressing a bug, remove the "Yes" from the "Regression" field and Save the bug
    1. Do not clear the "Baseline" field
  4. Any bug that has resurfaced, email us immediately

 

Notes:

  • Read the History of a bug. Many questions you will have will already be answered in the History section
  • Feel free to edit a bug or its test project if needed.
    • Keep the old info and projects.
